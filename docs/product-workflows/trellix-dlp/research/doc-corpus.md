## Capability: Policy Authoring

### Overall Workflow

The Trellix DLP policy authoring workflow follows a hierarchical build-up model. Before creating rules that protect data, you must first create the building blocks (definitions), then assemble them into classifications, build rules referencing those classifications, group rules into rule sets, assign rule sets to policies, and finally assign policies to systems via the ePO System Tree. [S5][S6][S7]

**High-level workflow (as documented):**
1. Create Definitions (reusable building blocks)
2. Create Classifications (using definitions as criteria)
3. Create Data Protection Rules (referencing classifications)
4. Group Rules into Rule Sets
5. Assign Rule Sets to Policies in the Policy Catalog
6. Assign Policies to systems via the ePO System Tree
7. Push policies to endpoints via agent communication

**Navigation root:** Menu > Data Protection > DLP Policy Manager [S23][S1]

### Level 1: Definitions

**Navigation path:** Menu > Data Protection > DLP Policy Manager > Definitions tab [S23]

Definitions are reusable building blocks used in both Classifications (content criteria) and Data Protection Rules (conditions). There are two contexts where definitions appear:

1. **DLP Policy Manager > Definitions** -- definitions usable in Rules (source/destination conditions)
2. **DLP Policy Manager > Classification > [classification name] > Definitions** -- definitions usable within classification criteria

**IMPORTANT:** Definitions under DLP Policy Manager can only be used in Rules. Definitions under the Classification menu can only be used under a classification. Some definition types (e.g., Advanced Patterns, Dictionaries) appear in both contexts. [S73 community post]

#### Definition Categories and Types

The DLP Policy Manager Definitions page organizes definitions into the following categories: [S23][S24]

**Source/Destination Definitions** (used in rule conditions):

| Definition Type | Description | Key Fields | Used In |
|----------------|-------------|------------|---------|
| **End-User Groups** | AD-based user/group definitions | Name, AD Groups, Individual Users; Requires registered LDAP server | Rules (sender/user conditions) |
| **Email Address Lists** | Lists of email addresses | Email addresses (individual or domain wildcards) | Email protection rules |
| **URL Lists** | Lists of web URLs | URLs, URL patterns | Web/cloud protection rules |
| **Network Definitions** | Network address ranges | IP addresses, IP ranges, subnets, CIDR notation | Network communication rules |
| **Network Port Definitions** | Port number definitions | Port numbers, port ranges | Network communication rules |
| **Network Printers** | Printer definitions | UNC path, IP address | Printer protection rules |
| **Application Templates** | Application identification | Product name, Vendor name, Executable file name, Window title; Can define single app or group | Application file access rules, all rule types |

**Content/Classification Definitions** (used in classifications):

| Definition Type | Description | Key Fields | Used In |
|----------------|-------------|------------|---------|
| **Advanced Patterns** | Regular expression patterns | Name, Description, Regex pattern, Validator (optional: Luhn, USPS, custom), Score, Required/Optional flag | Content classification criteria |
| **Dictionaries** | Collections of keywords/phrases | Name, entries (string + score), import from CSV/file | Content classification criteria |
| **Document Properties** | File metadata conditions | Property sets based on: Author, Keywords/Tags, Last Saved By, Subject, Title, custom properties | Content classification criteria |
| **File Extensions** | File extension matching | Extension strings (e.g., .docx, .pdf) | Rules and classifications |
| **True File Type** | Binary file type detection | True file type groups (detects renamed extensions) | Rules and classifications |

[S11][S12][S14][S15][S16][S17][S18][S19][S20][S21][S22][S23]

#### Advanced Patterns -- Detailed Configuration

**Navigation:** DLP Policy Manager > Definitions > Advanced Patterns (or Classification > [name] > add definition) [S11][S12][S13][S14]

**Fields:**
- **Name**: Descriptive name for the pattern
- **Description**: Free text
- **Matched expression (regex)**: The regular expression to match. Uses standard regex syntax with extensions [S14][S75]
- **Ignored expressions**: Optional regex patterns to exclude from matches
- **Validator**: Optional secondary validation:
  - Luhn algorithm (credit card checksum)
  - USPS checksum
  - Custom validator (script-based)
- **Score**: Numeric weight for this pattern when used in classification thresholds
- **Required/Optional**: Whether this pattern must match or contributes to scoring

**Built-in advanced patterns include:** Credit Card Number, Social Security Number, ABA Routing Number, Driver's License patterns (per-state), and many more organized by region/compliance. [S8][S13]

**Regex capabilities:** Standard PCRE-compatible regex. Supports character classes, quantifiers, alternation, groups, lookahead/lookbehind. [S14][S75]

#### Dictionary Definitions -- Detailed Configuration

**Navigation:** DLP Policy Manager > Definitions > Dictionaries (or Classification context) [S15][S16][S17]

**Fields:**
- **Name**: Dictionary name
- **Entries**: List of keyword/phrase strings, each with an associated score
- **Score**: Numeric weight per entry
- **Import capability**: Can import from CSV file or text file
- **Start with / End with / Contains**: Matching mode per entry

**Built-in dictionaries:** Medical terms, profanity, financial terms, legal terms, HIPAA-related terms, and more. [S8]

#### Application Templates -- Detailed Configuration

**Navigation:** DLP Policy Manager > Definitions > Application Templates [S21][S22]

**Fields:**
- **Name**: Template name
- **Operating System**: Windows / macOS
- **Application properties (one or more):**
  - Product Name (from file properties)
  - Vendor Name (from file properties)
  - Executable File Name (exact or wildcard)
  - Window Title (contains/equals string)
  - SHA-256 hash
- **Strategy**: Trusted / Monitored / Blocked
- **Can define:** Single application or group of similar applications

**Built-in templates:** Common browsers (Chrome, Firefox, Edge), email clients (Outlook), cloud apps (OneDrive, Dropbox, Google Drive), communication tools (Slack, Teams, Zoom), and more. [S22]

#### End-User Groups

**Navigation:** DLP Policy Manager > Definitions > End-User Groups [S23][S73]

**Prerequisites:** Registered LDAP server in ePO [S73]

**Fields:**
- **Name**: Group name
- **Source**: Active Directory Groups, Individual Users, or combination
- **AD Group selection**: Browse/search registered LDAP directories
- **Individual user**: SID or username specification

**Note:** DLP policies themselves cannot be assigned via user-based policy assignment rules in ePO. However, End-User Groups can be used within individual rules to scope rule applicability per-user. [S73]

#### Network Printer Definitions

**Navigation:** DLP Policy Manager > Definitions > Network Printers [S58]

**Fields:**
- **Name**: Printer name
- **UNC path**: Network path to the printer
- **IP address**: IP address of the network printer

### Level 2: Classifications

**Navigation path:** Menu > Data Protection > DLP Policy Manager > Classification tab [S10]

Classifications are the intelligence layer of Trellix DLP. They define WHAT constitutes sensitive data by combining definitions into criteria. Classifications are referenced by Data Protection Rules to trigger policy enforcement. [S9][S55]

#### Classification Tab Structure

The Classification page has the following sub-tabs: [S10]
1. **Content Classification Criteria** -- rule-based content matching
2. **Manual Classification** -- user-applied labels
3. **Register Documents** -- document fingerprinting
4. **Ignored Text** -- boilerplate exclusions (whitelisted text)
5. **Definitions** -- classification-scoped definitions (distinct from DLP Policy Manager definitions)

#### Content Classification Criteria

**Navigation:** Classification > Content Classification Criteria [S9][S10][S54]

A content classification is built from one or more **conditions** combined with AND/OR logic. Each condition references a definition type:

**Condition types available:**
- **Advanced Pattern**: Match regex patterns (with optional validator and score threshold)
- **Dictionary**: Match keyword/phrase entries (with score threshold)
- **Document Properties**: Match file metadata
- **True File Type**: Match binary file type
- **File Extension**: Match file extension
- **Keyword**: Simple keyword matching
- **Proximity**: Match patterns within N characters of each other
- **Exact Data Match (EDM)**: Match against fingerprinted structured data
- **Content Fingerprinting**: Match against fingerprinted document content

**Combining conditions:**
- **ANY (OR)**: Any single condition must match
- **ALL (AND)**: All conditions must match
- **Score threshold**: Cumulative score from weighted definitions must exceed threshold
- **Occurrence count**: Minimum number of matches required

**Creating a classification (task steps):** [S9]
1. In ePO, navigate to Menu > Data Protection > Classification
2. Click Actions > New Classification (or duplicate existing)
3. Enter Name and Description
4. On Content Classification Criteria tab, click the + icon to add criteria
5. For each criterion, select the definition type and configure:
   - Select or create the definition (e.g., an Advanced Pattern)
   - Set the condition operator (is, contains, matches)
   - Set score thresholds if applicable
6. Combine criteria with AND/OR operators as needed
7. Save the classification

**Built-in classifications:** PII classifications per country (US SSN, UK NIN, etc.), HIPAA, PCI-DSS, GDPR, financial data, legal documents, source code, and more. These are read-only but can be duplicated and customized. [S64]

#### Content Fingerprinting Criteria

**Navigation:** Classification > Content Classification Criteria > (add fingerprinting condition) [S54]

Content fingerprinting creates signatures of protected documents. When content matching these signatures appears in any data channel, the rule triggers.

**Steps to create content fingerprinting criteria:** [S54]
1. Navigate to Classification > select or create a classification
2. Add a Content Fingerprinting criterion
3. Configure the signature source (registered document repository)
4. Set the match percentage threshold (how much of the document must match)

#### Manual Classification

**Navigation:** Classification > Manual Classification [S49][S50][S51]

Manual classification enables end users to classify documents directly from within applications (Microsoft Office, email clients) by applying labels.

**Configuration fields:**
- **General Settings:**
  - Force classify on save: Require users to classify before saving
  - Force classify on send: Require classification before sending emails
  - Force classify on print: Require classification before printing
- **Classifications list**: Define which classification labels appear in the user's right-click / ribbon menu
- **Visual labels** (11.14.x+): Apply visible watermarks/headers/footers based on classification [S51]
- **Persistent tags**: Classification metadata embedded in the document that travels with the file across systems
- **Embedded tags**: Tags stored within document metadata properties

**End-user interface:**
- Right-click context menu in Windows Explorer
- Ribbon button in Microsoft Office applications
- Dialog prompt before save/send/print (if forced)

#### Register Documents

**Navigation:** Classification > Register Documents [S52][S53]

Registered documents allow fingerprinting of entire documents or document repositories for content matching.

**Configuration:**
- **Shared Storage Location**: UNC path (SMB) or WebDAV URL where fingerprints are stored
- **Document source**: File share path, specific files, or folder
- **Upload**: Manually upload documents for fingerprinting
- **Automatic registration**: Schedule periodic re-scanning of document repositories

**Fields on Register Documents page:** [S53]
- File/Folder path
- Recurse subfolders option
- Registration status
- Last scan date

#### Exact Data Matching (EDM)

**Navigation:** Classification criteria > Exact Data Match condition [S45][S46][S47][S48]

EDM protects structured data (database records) by fingerprinting the actual values in rows/columns.

**Two versions exist:**
1. **EDM Enhanced** (current) -- improved performance and accuracy [S45]
2. **EDM Old** (legacy) -- maintained for backward compatibility [S46]

**Configuration steps (EDM Enhanced):** [S45][S47][S48]
1. Export database records to CSV/TSV file
2. Prepare the fingerprint file using the **EDMTrain** tool
3. Upload fingerprint file to ePO
4. Create a classification with EDM criteria
5. Select columns to match (e.g., SSN + Name + DOB)
6. Set the match threshold (minimum columns that must match in same row)
7. Reference the classification in data protection rules

**Fingerprint file preparation:** [S48]
- Use the EDMTrain utility (command-line tool)
- Input: CSV or TSV file with headers
- Output: Fingerprint signature file (.edm)
- Upload to ePO server

### Level 3: Data Protection Rules

**Navigation path:** Menu > Data Protection > DLP Policy Manager > Rule Sets > [Rule Set Name] > Add Rule [S25][S56]

Data Protection Rules define the enforcement actions taken when classified sensitive data is detected in a specific data channel. Each rule type corresponds to a specific data loss vector. [S28][S29]

#### Nine Rule Types

| # | Rule Type | Vector Protected | Applies To | Key Conditions |
|---|-----------|-----------------|------------|----------------|
| 1 | **Email Protection** | Email (Outlook, SMTP) | Endpoint + Network (Prevent) | Sender, Recipient, Classification, Envelope/Header/Body, Attachments | [S30][S57]
| 2 | **Web Protection** | HTTP/HTTPS web posts | Endpoint + Network (Prevent) | URL destination, Classification, HTTP method | [S38]
| 3 | **Cloud Protection** | Cloud storage/apps (OneDrive, Dropbox, etc.) | Endpoint | Cloud service, Classification, User | [S36]
| 4 | **Removable Storage Protection** | USB drives, external media | Endpoint | Device type, Classification, User | [S31]
| 5 | **Network Share Protection** | Network file shares (SMB/CIFS) | Endpoint | Share path, Classification, User | [S35]
| 6 | **Network Communication Protection** | Network protocols (FTP, IM, custom) | Endpoint | Protocol/port, Application, Classification | [S32]
| 7 | **Clipboard Protection** | Copy/paste between applications | Endpoint | Source app, Destination app, Classification | [S33]
| 8 | **Printer Protection** | Print operations | Endpoint | Printer (local/network), Classification, User | [S34]
| 9 | **Application File Access Protection** | Application read/write of files | Endpoint | Application template, File location, Classification | [S37]

#### Common Rule Structure (All Rule Types)

Every data protection rule shares a common structure: [S28][S29][S56]

**Condition Tab:**
- **Classification**: Select one or more classifications (the "what" to protect)
- **End-User**: Optionally restrict rule to specific end-user groups
- **Source/Destination conditions**: Type-specific (e.g., sender/recipient for email, URL for web)
- **Application**: Optionally restrict to specific applications (via Application Templates)
- **File conditions**: File type, file extension, file size

**Reaction Tab:**
- **Action**: The enforcement action to take (see reactions matrix below)
- **User Notification**: Optional popup message to the end user
- **Report to ePO**: Whether to generate an incident in the DLP Incident Manager
- **Store Original Evidence**: Whether to capture a copy of the data for forensic review
- **Severity**: Critical / High / Medium / Low / Informational

**State:**
- **Enabled / Disabled**: Toggle rule enforcement

**Recipient Domain Threshold Settings:** [From S2 search results]
- Email protection rules can trigger different reactions based on recipient domain threshold counts

#### Email Protection Rule -- Detailed Fields [S30][S57]

**Applies to:** DLP Endpoint (Outlook integration) and DLP Prevent (SMTP gateway)

**Condition tab fields:**

### Level 6: Applicability / Assignment

**Navigation path:** Menu > Systems > System Tree > [select group/system] > Assigned Policies tab [S39][S40][S70][S71]

Policy assignment connects DLP policies to managed systems through the ePO System Tree.

#### ePO System Tree Structure [S71]

The System Tree is a hierarchical organization of all managed endpoints:
- **Root** (My Organization)
  - **Groups** (by location, department, type, etc.)
    - **Sub-groups** (nested hierarchy)
      - **Individual Systems** (endpoints)

**Policy inheritance:** Policies assigned at a parent group are inherited by all child groups and systems. A policy assigned at a lower level overrides the inherited policy. [S70][S71]

#### Assignment Process [S39][S40]

**Task: Assign and push a DLP policy to an Endpoint group:**
1. Navigate to Menu > Systems > System Tree
2. Select the target group or individual system
3. Click the Assigned Policies tab
4. Find the "Data Loss Prevention [version]" product row
5. Click "Edit Assignment" for the DLP Policy category
6. Select the desired policy from the dropdown
7. Choose "Break inheritance" if overriding a parent assignment
8. Click Save
9. To push immediately: Actions > Agent > Wake Up Agents > check "Send Policies"

**Task: Assign and push a DLP policy for Network (Prevent/Monitor):** [S40]
1. Navigate to Menu > Systems > System Tree
2. Select the DLP Prevent or Monitor appliance
3. Assign the DLP Policy as above
4. The network appliance will receive the policy on next agent-to-server communication interval (ASCI)

#### Agent Communication and Policy Delivery [S72]

**Agent-to-Server Communication Interval (ASCI):**
- The Trellix Agent on each endpoint communicates with the ePO server at regular intervals (default: 60 minutes)
- On each ASCI, the agent checks for policy updates and downloads any changes
- Administrators can force immediate policy delivery using "Wake Up Agents" [S72]

**Wake Up Agents:**
- **Navigation:** System Tree > [select systems] > Actions > Agent > Wake Up Agents
- **Options:** Send Policies, Run Client Tasks
- **Effect:** Forces immediate agent-to-server communication outside the ASCI schedule

**DLP Policy Apply via REST API:** [S41]
- REST API endpoint available to programmatically trigger policy application
- Uses ePO authentication (username/password)
- Can be integrated into automation workflows

#### Policy Assignment Rules (for system-based assignment)

**Navigation:** Menu > Policy > Policy Assignment Rules [S73]

**Important limitation:** DLP policies cannot be used with user-based policy assignment rules in ePO. Policy assignment is system-based only (by System Tree location). However, individual rules within a DLP policy can use End-User Groups to scope enforcement per-user. [S73]

## Key Findings

### 1. Hierarchical Policy Model
Trellix DLP uses a strict bottom-up hierarchy: Definitions -> Classifications -> Rules -> Rule Sets -> Policies -> System Tree Assignment. Each level depends on the lower levels being configured first. This is the core authoring workflow.

### 2. Dual Definition Contexts
Definitions exist in TWO different contexts within the DLP Policy Manager: those used in Rules (source/destination conditions) and those used in Classifications (content criteria). Some definition types appear in both contexts but are managed separately. This is a potential source of confusion for administrators.

### 3. Nine Distinct Rule Types
Each data loss vector (email, web, cloud, removable storage, network share, network communication, clipboard, printer, application file access) has its own dedicated rule type with vector-specific conditions. All share a common structure (Classification + Conditions + Reactions).

### 4. Shared Classification Engine
DLP Endpoint and DLP Prevent (network) share a common classification engine, enabling administrators to create a single email/web policy that enforces consistently across endpoints and the network gateway. This is a significant architecture advantage.

### 5. EDM Has Two Implementations
Exact Data Matching exists in both an "Enhanced" (current) and "Old" (legacy) implementation. New deployments should use EDM Enhanced. The fingerprint preparation workflow requires the external EDMTrain utility.

### 6. Policy Assignment is System-Based Only
DLP policies are assigned to systems via the ePO System Tree, NOT to users. User-based policy assignment rules do NOT work with DLP. User-scoping is achieved within individual rules by referencing End-User Groups (Active Directory-based).

### 7. REST API Available for Automation
REST API endpoints exist for applying policies, backing up configurations, importing definitions, and retrieving incidents. Authentication uses ePO credentials. Sample code is available in Java and PowerShell.

### 8. Pre-Built Templates Accelerate Deployment
Out-of-the-box rule set templates for GDPR, HIPAA, PCI-DSS, SOX, NIST, ISO 27001, and SOC 2 provide immediate DLP coverage that can be customized.

### 9. Manual Classification with Visual Labels (11.14.x)
The latest version (11.14.x) adds visual labels (watermarks, headers, footers) to manual classification, enhancing the user-facing experience of data tagging.

### 10. Agent Communication Controls Policy Delivery
Policy changes are delivered to endpoints via the Trellix Agent's ASCI (Agent-to-Server Communication Interval). Administrators can force immediate delivery using "Wake Up Agents" from the System Tree.
