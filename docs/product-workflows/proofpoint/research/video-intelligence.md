## Tribal Knowledge Summary

### Prerequisites Often Missed

1. URL Defense must be explicitly enabled after provisioning — it is OFF by default. Navigate to Administration > Account Management > Features. Mentioned as a common omission in Video 5 and confirmed in community documentation.

2. TAP VAP list must be manually imported into Isolation policy — automatic sync does not exist. Administrators must re-import the VAP list after any TAP threat review cycle that changes the VAP roster.

3. Policy routes must be defined in PPS before Email Firewall rules can correctly scope traffic. Rule conditions referencing "default_inbound" assume this route exists; custom environments may have renamed it.

4. Behavioral AI in Adaptive Email DLP requires a learning / warm-up period before enforcement yields accurate detections. Enforcing block actions before the model has trained will produce high false positive rates.

### Common Configuration Mistakes

1. Adding Email Firewall rules without specifying a Route condition — rules then apply to all routes including outbound. Source: community documentation cross-referencing Video 2 at ~2:00.

2. Setting DLP / encryption filter Scope to Group or User and expecting the Encrypt action to appear — Encrypt is only available at Company + Outbound scope. Source: Video 7 at ~2:00.

3. Testing rules immediately after saving without waiting for propagation (5–30 min). Source: Video 2 at ~3:00.

4. Deploying new filter policies directly at Company scope without testing at User scope first, causing organization-wide mail disruption. Source: Video 20 at ~4:30.

5. Configuring spam aggressiveness in Filter Policies and assuming Spam Settings are also updated — the two UIs are independent. Source: Video 21 at ~1:00.

### Workarounds and Shortcuts

1. To approximate per-group email encryption (since Encrypt action requires Company scope), create a Company-scope outbound encryption filter with recipient domain exceptions for everyone outside the target group. Source: inferred from Video 7 at ~2:00.

2. Use "Tag Subject" as a secondary action on DLP filters to add a header tag before enforcement — allows a passive monitoring phase without quarantine. Source: Video 20 at ~3:00 (implied by testing workflow).

3. HTML attachment blocking: pull 30-day inbound mail logs to enumerate all senders sending HTML attachments before enabling the rule, and pre-populate the exception list. Source: Video 20 at ~2:30.

4. For ITM rule testing, set primary action to "Allow + Log Only" mode (low severity, notification only) for 1–2 weeks before activating block actions. Source: Video 16 workflow pattern.

### Version-Specific Notes

1. Proofpoint Essentials UI refresh (2023) introduced cleaner visuals and faster load times. Filter policy and spam settings navigation paths documented above reflect the post-2023 UI; pre-2023 videos (Videos 8–13) show an older interface with slightly different visual layout but the same underlying navigation hierarchy.

2. Videos 3, 4, 9, 12, 13 (2017–2018) reference PPS on-premises admin interface. The navigation path System > Policy Route is PPS-specific and does not appear in Proofpoint Essentials or Proofpoint on Demand (PoD) cloud console.

3. Adaptive Email DLP (Video 22, January 2025) represents a behaviorally distinct DLP approach from the rule-based filter policy DLP available in all prior videos — the two coexist but use separate configuration interfaces.

4. TAP URL Isolation for VAPs (Video 17, 2019) predates the generalized Browser Isolation product (Video 18, 2022). By 2022 the product is rebranded to "Proofpoint Isolation" and accessed separately from TAP Dashboard.

### Comment-Sourced Intelligence

No comment-level intelligence was extractable for these videos. YouTube page rendering returns JavaScript configuration only; comment scraping was not possible via WebFetch. The following observations come from community article cross-references found during search:

1. Community article (proofpoint.my.site.com): The anti-spoof rule in PPS must be explicitly enabled — it is not active in default PPS deployments. Access via Email Firewall > Rules, find the anti-spoof rule, set to On. This is not called out in any standard tutorial video.

2. Community article (proofpoint.my.site.com — Best Practices for Tuning Spam Module Rules): Spam module rules should be tuned incrementally — start with the highest-confidence rules (score 100) and monitor false positives before reducing the threshold. Aggressive tuning in a single pass is a documented common mistake.

3. Community article (proofpoint.my.site.com — TAP: How to Exempt a Sender from TAP Alerts): TAP alerts for specific senders can be suppressed via the TAP Dashboard exemption list — this is separate from safe-sender lists in the email protection interface. Admins who whitelist a sender only in the Email Protection safe-sender list will still receive TAP dashboard alerts for that sender.
