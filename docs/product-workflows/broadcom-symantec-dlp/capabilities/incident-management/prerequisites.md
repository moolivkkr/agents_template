# Prerequisites: Incident Management

> **Applies to:** Broadcom Symantec DLP 16.0 through 26.1
> **Purpose:** Everything that must be in place before incidents flow and analysts can triage them

---

## Infrastructure Requirements

### 1. Enforce Server

| Requirement | Details |
|-------------|---------|
| Enforce Server | Installed, running, accessible via web console |
| Oracle Database | Backing the Enforce Server -- stores all incidents, evidence, audit trail |
| Database Sizing | Plan for incident volume: ~1-5 KB per incident metadata; 50-500 KB with evidence; 500K+ incidents/year for mid-size enterprise |
| SMTP Configuration | Required for email notifications (response rules, alerts, end user remediation) |
| Console URL | `https://enforce-server:443/ProtectManager` -- must be accessible to all analysts |

### 2. At Least One Detection Server

Incidents are generated by detection servers. You need at least one:

| Detection Server | Generates Incidents For |
|-----------------|------------------------|
| Network Monitor Server | Passively observed network traffic (email, web, FTP) |
| Network Prevent for Email | Email messages that violate policies |
| Network Prevent for Web | Web uploads/posts that violate policies |
| Endpoint Prevent Server | Endpoint activities (USB, clipboard, print, browser) |
| Network Discover Server | Files found on shares, databases, SharePoint |
| Cloud Detector (CDS/DDS) | Cloud app activity (O365, Google, Box) |

### 3. At Least One Active Policy

No policy = no incidents. Verify:
```
Enforce Console > Manage > Policies > Policy List
```
- At least one policy must be **Enabled**
- Policy must be in a **Policy Group** assigned to a running detection server
- Policy must have at least one **Detection Rule** (content match, data identifier, etc.)

## User Roles for Incident Management

### Role Configuration

**Navigation:** System > Login Management > Roles

DLP uses Role-Based Access Control (RBAC). The following privileges are relevant to incident management:

| Privilege | Description | Who Needs It |
|-----------|-------------|-------------|
| View Incidents | See incidents in the incident queue | All analysts |
| Edit Incidents | Change status, add notes, update attributes | All analysts |
| View Masked Data | See full matched content (unmasked) | Senior analysts, investigators |
| Execute Smart Response | Trigger Smart Response rules | Trained analysts |
| View Reports | Access saved reports and dashboards | Analysts, managers |
| Create/Edit Reports | Create and modify saved searches | Senior analysts, managers |
| Policy Administration | Create/edit policies (needed to tune policies based on false positives) | Policy admins |
| Incident Reporting API | Access the REST API for incident operations | SOAR systems, automation accounts |

### Minimum Roles for an Incident Management Team

| Role | Privileges | Assigned To |
|------|-----------|------------|
| DLP Analyst | View Incidents, Edit Incidents, Execute Smart Response, View Reports | SOC analysts, DLP responders |
| DLP Senior Analyst | All Analyst privileges + View Masked Data, Create/Edit Reports | Senior team members, investigators |
| DLP Policy Admin | All Senior Analyst privileges + Policy Administration | Policy owners |
| DLP API Service Account | Incident Reporting API, View Incidents, Edit Incidents | SOAR platform, automation scripts |

### Creating the Analyst Role

1. Navigate to **System > Login Management > Roles**
2. Click **Add Role**
3. Name: "DLP Analyst"
4. Enable privileges:
   - [x] View Incidents
   - [x] Edit Incidents
   - [x] Execute Smart Response
   - [x] View Reports
5. Under **Incident Access**: Define which incidents this role can see
   - Option: All incidents
   - Option: Only incidents matching specific policies or policy groups
6. Click **Save**
7. Assign users to this role

## Notification Infrastructure

### SMTP Configuration (Required for Notifications)

Response rules and end user remediation require email sending capability:

1. Navigate to **System > Settings > General**
2. Configure SMTP settings:
   - SMTP Server: `smtp.corp.local`
   - Port: 25 (or 587 for TLS)
   - From Address: `dlp-noreply@corp.com`
   - Authentication: If required by your SMTP server
3. Send test email to verify

### Syslog Server (Required for SIEM Integration)

If using response rules that log to syslog:

1. Syslog server must be reachable from the Enforce Server
2. Protocol: UDP (port 514) or TCP (port 514 or custom)
3. Format: CEF or plain syslog (configurable per response rule)

## Custom Attributes (Recommended)

Define custom attributes before incidents start flowing:

**Navigation:** System > Incident Data > Attributes > Custom Attributes tab

Recommended custom attributes:

| Attribute Name | Type | Values | Purpose |
|---------------|------|--------|---------|
| Remediation Owner | Text | Free text | Track who is responsible |
| Cost Center | Dropdown | List of cost centers | Map to business unit |
| Risk Assessment | Dropdown | Critical, High, Medium, Low, Acceptable | Analyst risk classification |
| Regulatory Scope | Multi-select | PCI, HIPAA, GDPR, SOX | Compliance mapping |
| Resolution Category | Dropdown | Policy Violation, False Positive, Exception, Training | Categorize resolutions |

### Custom Incident Statuses (Recommended)

Add statuses beyond the defaults:

**Navigation:** System > Incident Data > Attributes > Incident Statuses tab

Recommended additions:

| Status | Purpose |
|--------|---------|
| Waiting for User Response | Sent to data owner, awaiting reply |
| Under Legal Review | Escalated to legal |
| Training Assigned | User enrolled in security awareness |
| Exception Granted | Approved business exception |
| Pending Manager Approval | Awaiting manager sign-off |

## Lookup Plugins (Recommended)

Lookup plugins auto-populate custom attributes with data from external sources:

### LDAP Lookup Plugin

Automatically enrich incidents with user details from Active Directory:

1. **Create directory connection**: System > Settings > Directory Connections
   - Server: `ldap.corp.local`
   - Port: 389 (or 636 for LDAPS)
   - Base DN: `dc=corp,dc=local`
   - Bind credentials: Service account with LDAP read access
2. **Create LDAP plugin**: System > Lookup Plugins > New Plugin > LDAP
3. **Map attributes**:
   ```
   attr.Department = ou=Users,dc=corp,dc=local:(sAMAccountName=$sender.login$):department
   attr.Manager = ou=Users,dc=corp,dc=local:(sAMAccountName=$sender.login$):manager
   attr.CostCenter = ou=Users,dc=corp,dc=local:(sAMAccountName=$sender.login$):extensionAttribute1
   ```
4. **Verify**: Custom attribute names must match those defined in System > Incident Data > Attributes

## SOAR Integration Prerequisites (Optional)

### For Cortex XSOAR

| Requirement | Details |
|-------------|---------|
| Cortex XSOAR instance | Running and accessible |
| Symantec DLP v2 integration pack | Installed from XSOAR Marketplace |
| DLP API service account | User with "Incident Reporting API" privilege |
| Network connectivity | XSOAR server must reach Enforce Server on port 443 |

### For ServiceNow

| Requirement | Details |
|-------------|---------|
| ServiceNow instance | Running (any supported version) |
| Symantec DLP integration app | Installed from ServiceNow Store |
| DLP API service account | User with API access |
| Bidirectional connectivity | ServiceNow <-> Enforce Server on port 443 |

### For Generic SOAR/API

| Requirement | Details |
|-------------|---------|
| API user account | DLP user with "Incident Reporting API" role |
| Network access | SOAR platform must reach `https://enforce-server:443` |
| TLS certificate | Enforce Server TLS certificate must be trusted by SOAR platform |

## Pre-Launch Checklist

Before going live with incident management:

- [ ] Enforce Server is running and accessible via web console
- [ ] At least one detection server is registered and running
- [ ] At least one detection policy is enabled and deployed
- [ ] Analyst roles are created with appropriate privileges
- [ ] Analyst user accounts are created and assigned to roles
- [ ] SMTP is configured for email notifications (test email sent successfully)
- [ ] Custom incident statuses are defined (if needed beyond defaults)
- [ ] Custom attributes are defined (Department, Cost Center, etc.)
- [ ] LDAP lookup plugin is configured (auto-populate user info on incidents)
- [ ] Smart Response rules are created (at least: Mark False Positive, Escalate, Close Resolved)
- [ ] Syslog server configured (if SIEM integration required)
- [ ] SOAR integration tested (if using XSOAR, ServiceNow, etc.)
- [ ] Incident review process documented (who reviews, SLA, escalation path)
- [ ] Database sizing verified (Oracle tablespace for incident storage)
- [ ] Incident retention policy defined (how long to keep incidents and evidence)
