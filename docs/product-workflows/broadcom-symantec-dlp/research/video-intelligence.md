## 2. Per-Capability Workflow Extractions

### 2.1 Installation & Architecture

**Source videos:** #9-12 (Deploy series), #5, #10, #11

#### Installation Order (CRITICAL -- must follow this sequence)
1. **Install Oracle Database** (Video #9) -- Oracle 19c required for DLP 16.0+
2. **Install Enforce Server** (Video #10) -- Central management console; connects to Oracle DB
3. **Install Detection Servers** (Video #11) -- Discovery and Endpoint Protection servers
4. **Install Endpoint Agents** (Video #12) -- Deploy to Windows/Mac/Linux endpoints
5. **Configure Basic Policies** (Video #12) -- Keyword-based policy as first validation

#### Architecture: Component Communication
```
                    +-------------------+
                    |   Enforce Server   |
                    |  (Management Hub)  |
                    +--------+----------+
                             |
              +--------------+--------------+
              |              |              |
    +---------v--+   +------v-----+   +----v--------+
    | Network    |   | Network    |   | Endpoint    |
    | Monitor    |   | Prevent    |   | Prevent     |
    | (passive)  |   | (inline)   |   | Server      |
    +------------+   +---+--------+   +------+------+
                         |                    |
                    +----v----+          +----v----+
                    | Email   |          | DLP     |
                    | Prevent |          | Agents  |
                    +---------+          +---------+
```

- **Enforce Server**: Central management; hosts policies, incidents, reports, user management
- **Detection Servers**: Process content inspection; multiple types (Network Monitor, Network Prevent for Email, Network Prevent for Web, Endpoint Prevent, Network Discover)
- **DLP Agents**: Installed on endpoints; communicate with Endpoint Prevent Server via TLS
- **Communication**: Enforce <-> Detection Servers use TLS 1.3 (default in recent versions); Agents <-> Endpoint Server use certificate-based authentication

#### Architecture Best Practices (from search results)
- Place 2 Endpoint Servers in corporate LAN + 2 in DMZ
- Agents on LAN connect to LAN Endpoint Server with failover to second
- Remote agents directed to DMZ Endpoint Servers via load balancer
- Load balancer must use "Source IP persistence" set to 24 hours
- All Endpoint Servers connect back to the Enforce Server

### 2.2 Policy Creation & Authoring

**Source videos:** #14, #16, #17, #18

#### Navigation Path
```
Manage > Policies > Policy List > New Policy
```

#### Two Approaches to Policy Creation
1. **From Template** -- Select from 65+ built-in templates (PCI, HIPAA, GLBA, SOX, etc.)
   - Navigate: `Manage > Policies > Policy List > New Policy > Template List`
   - Select template > Click Next > Configure > Set policy group > Enable/Disable
   - Default status: "Test Without Notifications" (safe for initial rollout)

2. **From Scratch** -- Build custom policy with detection rules
   - Navigate: `Manage > Policies > Policy List > New Policy > Create New Policy`
   - Define policy name and description
   - Add detection rules (one or more)
   - Add exceptions (optional)
   - Assign to policy group
   - Set response rules

#### Detection Rule Types (Policy Conditions)
| Rule Type | Description | Use Case |
|-----------|-------------|----------|
| Content Matches Keyword | Keyword or phrase matching | Simple data classification |
| Content Matches Regular Expression | Regex pattern matching | Account numbers, IDs |
| Content Matches Data Identifier | System or custom data identifiers | SSN, credit cards, IBAN |
| Content Matches Exact Data | EDM fingerprint matching | Customer databases |
| Content Matches Indexed Documents | IDM fingerprint matching | Confidential documents |
| Message/Component Matches | Envelope/header matching | Email-specific rules |
| Sender/Receiver Matches Pattern | User identity matching | Role-based exceptions |
| Endpoint Detection | Device and location matching | USB, print, clipboard |

#### Policy Tuning Best Practices (Video #17)
- Start with "Test Without Notifications" mode
- Monitor false positive rate before enabling enforcement
- Use exceptions to whitelist known-good patterns
- Review policy tuning reports regularly
- Staged rollout: Monitor > Notify > Block

### 2.3 Exact Data Matching (EDM)

**Source videos:** #19, #22 | **Cross-reference:** Broadcom TechDocs 16.0

#### What EDM Does
Fingerprints structured data sources (databases, CSV files, directory servers) to detect exact matches of sensitive records -- even partial records.

#### EDM Configuration Workflow
1. **Prepare Data Source**
   - Export data to delimited file (CSV, TSV) OR connect to database/LDAP directly
   - Columns represent data fields (Name, SSN, Email, etc.)
   - Clean data: remove headers if needed, ensure consistent formatting

2. **Create Exact Data Profile**
   - Navigate: `Manage > Data Profiles > Exact Data > Add Exact Data Profile`
   - Name the profile
   - Upload or connect to data source
   - Map columns to field types
   - Set Error Threshold (default 5% -- indexing stops if errors exceed this)

3. **Index the Data**
   - System extracts text, normalizes, and creates nonreversible hash
   - Indexing can be scheduled (daily, weekly) for auto-refresh
   - Index stored on Enforce Server, distributed to Detection Servers

4. **Create Policy Rule Using EDM**
   - Add new policy or edit existing
   - Add rule: "Content Matches Exact Data"
   - Select the Exact Data Profile
   - Configure matching: specify which fields to match and minimum field count
   - Optionally set "Match on X of Y fields" for partial matching

5. **Deploy and Test**
   - Assign policy to policy group
   - Start in "Test" mode
   - Validate detection on sample data

#### EDM Gotchas
- Error Threshold: If empty cells, wrong-type data, or extra cells exceed 5% of rows, indexing fails
- Large data sources (millions of rows) require careful scheduling to avoid performance impact
- Index must be refreshed when source data changes -- stale indexes miss new records
- EDM profiles with special upgrade path considerations when upgrading DLP versions

### 2.4 Indexed Document Matching (IDM)

**Source videos:** #22 | **Cross-reference:** Broadcom TechDocs 16.0

#### What IDM Does
Fingerprints unstructured documents (Word, PDF, JPEG, CAD, multimedia) to detect exact or partial copies, even if content is copy-pasted into a different document.

#### IDM Configuration Workflow
1. **Prepare Source Documents**
   - Gather all confidential documents to protect
   - Supported formats: Microsoft Office, PDF, binary files (JPEG, CAD, multimedia)
   - Organize into directories for bulk indexing

2. **Create Document Profile**
   - Navigate: `Manage > Data Profiles > Indexed Documents`
   - Create new profile
   - Upload documents or point to file share/directory
   - System fingerprints content (not just file hashes -- content-level fingerprinting)

3. **Configure Matching**
   - Full document match vs. partial match
   - Partial matching: detects fragments, percentages, or specific sections
   - Enable "IDM support for endpoints" for endpoint partial matching

4. **Create Policy Rule**
   - Add rule: "Content Matches Indexed Documents"
   - Select the Document Profile
   - Set sensitivity threshold

5. **CloudSOC IDM (Cloud)**
   - Use Remote Indexer Tool to convert IDM data source into remote index file
   - Upload remote index file to CloudSOC
   - Add IDM Index to a DLP Profile in CloudSOC

#### IDM Insights
- IDM detects "derived" content -- text copied from a source document to another file
- Catches copy-paste evasion attempts (employee copies portions of confidential plan to email)
- Partial matching is key for real-world detection -- users rarely copy entire documents

### 2.5 Vector Machine Learning (VML)

**Source videos:** #20 | **Cross-reference:** Broadcom TechDocs 16.0, VML Best Practices Guide

#### What VML Does
Statistical analysis to detect content similar to training examples. No need to locate and fingerprint all data -- train the system to learn what "looks like" confidential content.

#### VML Configuration Workflow
1. **Prepare Training Sets**
   - **Positive examples**: Documents you want to protect (e.g., financial reports, source code)
   - **Negative examples**: Documents that should NOT trigger (e.g., marketing materials)
   - Minimum recommended: 50+ documents per category for statistical significance
   - Quality > quantity -- representative examples are crucial

2. **Create VML Profile**
   - Navigate: `Manage > Data Profiles > Vector Machine Learning`
   - Create new profile
   - Upload positive training set
   - Upload negative training set

3. **Train the Profile**
   - System analyzes statistical patterns in positive vs negative examples
   - Review accuracy score
   - Iterate: add more training documents if accuracy is low

4. **Accept the Profile**
   - Review training results
   - Accept if accuracy meets threshold
   - Profile is ready for policy use

5. **Add to Policy**
   - Create or edit policy
   - Add detection rule referencing VML profile
   - Deploy to detection servers

#### VML Best Practices
- Use diverse, representative training documents
- Balance positive and negative sets
- Re-train periodically as content patterns evolve
- VML Best Practices Guide available at: Symantec_DLP_15.5_VML_Best_Practices_Guide.pdf / Symantec_DLP_15.8_VML_Best_Practices_Guide.pdf

### 2.6 Form Recognition

**Source videos:** #21 (DLP 14.5)

#### What Form Recognition Does
Detects sensitive information in scanned forms, images of filled-out forms (W-2s, applications, medical forms).

#### Workflow
1. Upload blank form templates as training data
2. System learns the form layout and field positions
3. When a filled-out version is detected, it triggers a policy match
4. Works with scanned images, not just digital forms

### 2.7 Response Rules Configuration

**Source videos:** #23 | **Cross-reference:** Broadcom TechDocs 16.0

#### Navigation Path
```
Manage > Policies > Response Rules > Add Response Rule
```

#### Response Rule Types
1. **Automated Response** -- Executes automatically when server evaluates incidents
2. **Smart Response** -- Authorized user executes from Incident Snapshot screen

#### Endpoint Prevent Response Actions

| Action | Behavior | Configuration |
|--------|----------|---------------|
| **Block** | Blocks data movement + optional notification | Set notification text, customize popup |
| **Notify** | Displays notification only (no blocking) | Configure notification content |
| **User Cancel** | Time-sensitive prompt -- user decides to proceed or stop | Set timeout duration; if no response, auto-blocks |
| **Encrypt** | Encrypts the content using configured encryption provider | Configure encryption settings |

#### Other Response Actions
- **Log to Syslog Server** -- Configure Host, Port, Message, Protocol (UDP/TCP), Level
- **Set Status** -- Automatically set incident status
- **Quarantine** -- Move file to quarantine location (Network Discover)
- **Copy to File Share** -- Copy detected file to investigation share
- **Apply Classification Label** -- Apply MIP/AIP classification label

#### Response Rule Configuration Steps
1. Click "Add Response Rule"
2. Choose Automated or Smart Response
3. Click Next to configure
4. Optionally add Conditions (when to execute -- if none, always executes on match)
5. Add at least one Action (required for validity)
6. Each action is evaluated independently
7. Save and associate with policies

#### Endpoint Notification Customization
- Custom popup text and branding
- Support for HTML in notification messages
- Localization for multi-language deployments
- Reference: KB article 159522 "Customizing the DLP Endpoint notification popup"

### 2.11 Cloud & CASB Integration

**Source videos:** #8, #34, #35, #36, #38

#### Symantec DLP Cloud Architecture
- Combines CASB (CloudSOC) + DLP Cloud Detection Service (CDS)
- No need to transfer data from cloud to on-premises appliance
- Files scanned in the cloud -- reduces latency and network traffic
- OCR in cloud extracts text from images for DLP policy application

#### Cloud Detection Service (CDS)
- Hosted service for DLP content inspection and remediation
- CloudSOC Securlet policies, Enforce-managed policies, and DLP Profile policies send content to CDS
- DLP Detectors scan for sensitive data
- Results flow back to CloudSOC or Enforce console

#### Cloud Managed DLP (CloudSOC Console)
- Single console for managing all CASB data channel DLP policies
- No need for separate Enforce Server for CASB use cases
- Provisioning: `CloudSOC Admin > DLP Configuration > Enable Cloud Detection Service`

#### Generative AI Protection (Videos #35, #36)
- Monitors data submitted to GenAI tools (ChatGPT, etc.)
- Applies DLP policies to content sent to AI services
- Block or warn users when sensitive data is being submitted to AI

### 2.12 Microsoft Information Protection (MIP) Integration

## 3. Tribal Knowledge NOT in Official Docs

These insights come from practitioner videos, community posts, and real-world deployment experience shared in the video content:

### 3.1 Deployment Strategy
- **Staged rollout is essential**: Deploy agents in phases, not all at once. Start with IT department for validation, then expand.
- **Monitor before enforce**: The single most common mistake is enabling blocking policies on day 1. Always start with "Test Without Notifications" or "Test With Notifications" before switching to "Block."
- **Political backlash risk**: Combining new endpoint agents with intrusive enforcement actions creates "grumpy employees and political backlash" (directly quoted from best practices content).

### 3.2 Detection Server Placement
- **DMZ servers are critical**: Remote/VPN users need DMZ-facing Endpoint Servers. Without them, agents queue incidents locally and only sync when on corporate network.
- **Load balancer gotcha**: Must set "Source IP persistence" to 24 hours on the load balancer. Without this, agents may report to different servers causing split incident data.

### 3.3 EDM Index Management
- **Stale indexes are silent failures**: If the data source changes (new employees, new records) but the index is not refreshed, those new records will not be detected. Set up automated re-indexing schedules.
- **Error threshold surprise**: The 5% default error threshold catches many admins off guard. Poorly formatted source data causes indexing to fail silently.

### 3.4 Policy Tuning
- **"Cherry-picking" advanced detection is a mistake**: Some organizations only use basic keyword matching when they should be using EDM, IDM, or VML for better accuracy and fewer false positives.
- **Policy exceptions decay over time**: Exceptions added for specific business needs should be reviewed periodically -- they may no longer be relevant.
- **Disabled policies leave gaps**: Any time a policy is disabled, there is a detection gap that may need to be filled by another policy.

### 3.5 Endpoint Performance
- **Local drive monitoring causes performance issues**: Enabling monitoring of every file created on local drives is a known performance killer. Be selective about what channels to monitor.
- **Agent package build determines connectivity**: The identification type (IP, hostname, FQDN) specified when building the agent package determines how the agent connects. Changing this later requires a new agent build.

### 3.6 ServiceNow Integration
- **Bidirectional sync is the key differentiator**: Most organizations start with one-way (DLP to ServiceNow) but the real value comes from bidirectional sync where ServiceNow status changes flow back to DLP.
- **End User Remediation (DLP 15.8+)**: Built-in solution for decentralized incident remediation -- assign incidents to data owners, not just security analysts.
