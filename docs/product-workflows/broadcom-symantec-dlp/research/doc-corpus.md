## 2. Product Architecture

### 2.1 Core Architecture [S1, S2, S4, S22]

Symantec DLP uses a hub-and-spoke architecture:

**Enforce Server (Hub)**
- Central management platform for the entire DLP deployment
- Web-based administration console (browser UI)
- Policy management, incident management, reporting, user/role administration
- Backed by an Oracle database (or embedded database for smaller deployments)
- Single Enforce Server manages all detection servers

**Detection Servers (Spokes)**
- Distributed processing engines that perform actual content inspection
- Multiple types, each covering a different data channel
- Register with and report to the Enforce Server
- Receive policy deployments from Enforce Server via policy groups

**DLP Agents (Endpoints)**
- Lightweight software installed on Windows, macOS, and Linux endpoints
- Communicate with Endpoint Servers at 15-minute intervals
- Perform local content inspection and enforcement
- Support both monitoring and prevention modes

### 2.2 Deployment Tiers [S1, S4]

| Tier | Description | Use Case |
|------|-------------|----------|
| Single-Tier | Database + Enforce + Detection on same machine | Testing/lab only |
| Two-Tier | Database + Enforce on one machine; detection servers separate | Small deployments |
| Three-Tier | Database, Enforce, Detection each on separate machines | Production (recommended) |

### 2.3 Detection Server Types [S1, S2, S3]

| Server Type | Data State | Function |
|-------------|-----------|----------|
| Network Monitor Server | Data in Motion | Passive monitoring of network traffic via span port |
| Network Prevent for Email Server | Data in Motion | Active SMTP inspection; block/redirect/quarantine email |
| Network Prevent for Web Server | Data in Motion | ICAP-based web proxy integration; block/allow HTTP/FTP |
| Network Discover Server | Data at Rest | Scans file shares, databases, SharePoint, Exchange |
| Cloud Storage Discover Server | Data at Rest | Scans cloud storage (Box, etc.) |
| Endpoint Prevent Server | Data in Use | Manages endpoint agents; monitoring + prevention |
| Cloud Detector | Data in Motion/Rest | Cloud-hosted detection (CDS) |

## 3. Detection Technologies

### 3.1 Described Content Matching (DCM) [S1, S4, S8]

DCM detects content by looking for matches on specific keywords, regular expressions, patterns, data identifiers, and file properties.

**Sub-capabilities:**

- **Keyword Matching**: Match on specific words or phrases; supports proximity matching and case sensitivity
  - Navigation: Policy > Rules > Add Rule > Content Matches Keyword
  - Supports: exact match, case-sensitive, proximity to other terms

- **Regular Expression Matching**: Full regex pattern matching against content
  - Navigation: Policy > Rules > Add Rule > Content Matches Regular Expression

- **Data Identifiers**: Pre-built algorithms combining pattern matching with validators
  - 30+ built-in identifiers (credit card with Luhn check, SSN, etc.)
  - Custom data identifiers supported
  - Patterns + validators reduce false positives
  - Navigation: Policy > Rules > Add Rule > Content Matches Data Identifier

- **File Property Matching**: Match on file name, type, size, creation date
  - Over 330 file types recognized by binary signature
  - Custom file types supported

- **Protocol/Destination Matching**: Match based on protocol (SMTP, HTTP, FTP, IM) or destination

### 3.2 Exact Data Matching (EDM) [S1, S4]

EDM detects structured, tabular data from databases or CSV/spreadsheet sources.

**How it works:**
1. Admin creates an Exact Data Profile
2. System indexes structured data source (CSV, database export)
3. Indexing process: extract text -> normalize -> non-reversible hash
4. Policy uses "Content Matches Exact Data" condition
5. At detection time, content is compared against hashed index

**Configuration:**
- Navigation: Manage > Data Profiles > Exact Data Profiles
- Must select at least 2-3 fields; one must be unique (e.g., SSN, credit card)
- Remote EDM Indexer available for off-server indexing
- Supports scheduling of index refreshes
- Field mapping checked against policy template suggestions

**Use cases:** PII protection (SSN, credit card, bank account), customer/employee records, price lists

### 3.3 Indexed Document Matching (IDM) [S1, S4]

IDM detects unstructured document content using fingerprinting.

**How it works:**
1. Admin creates an Indexed Document Profile
2. Documents are fingerprinted using rolling hashes
3. Supports partial matching (fragments, percentages, sections)
4. Detects "derived" content (copy/pasted from source documents)
5. Binary stamp matching for exact document detection

**File type coverage:**
- Microsoft Office documents
- PDFs
- Binary files (JPEGs, CAD designs, multimedia)
- Custom file types

**Configuration:**
- Navigation: Manage > Data Profiles > Indexed Document Profiles
- Remote Indexer Tool for off-server indexing
- Partial matching threshold configurable

### 3.4 Vector Machine Learning (VML) [S1, S4, S7]

VML uses statistical analysis to protect unstructured data by determining similarity to trained content.

**How it works:**
1. Admin prepares positive and negative training documents
2. Best practice: 500 total (250 positive, 250 negative); minimum 50 per set
3. System trains a statistical model
4. Model creates a VML Profile
5. Policy uses VML Profile for detection

**Use cases:** Financial reports, source code, intellectual property with subtle characteristics

**Configuration:**
- Navigation: Manage > Data Profiles > Vector Machine Learning Profiles
- Training set quality directly impacts accuracy
- VML Best Practices Guide available (S7)

### 3.5 Content Classifiers / Form Recognition [S1, S4]

- **Data Classifiers**: Detect content by image type or identifiable context
- **Form Recognition**: Detect forms (tax forms, medical forms, insurance forms)
- Pre-classifier inspects images to determine OCR suitability

### 3.6 Optical Character Recognition (OCR) [S1, S4]

- Extracts text from images (scanned documents, screenshots, pictures)
- Enables text-based detection rules on image content
- Available on-premises and in cloud (CASB/REST, EMAIL, WSS)
- Part of Sensitive Image Recognition add-on

### 3.7 User Risk-Based Detection [S2, S25]

- Integrates with Information Centric Analytics (ICA)
- Risk scores from 1-100 per user based on behavioral analysis
- Policy conditions can include risk score thresholds
- Example: "Trigger if user copies <10 SSNs to USB AND risk score >90"
- Navigation: Policy > Rules > User Risk condition

### 3.8 Directory Group Matching (DGM) [S1, S2]

- Detection rules based on sender/user identity
- Integrates with Active Directory and LDAP
- Sender/User based on Directory Server Group condition
- Microsoft Entra ID support (DLP 25.1+)
- Navigation: Policy > Rules > Sender/User Based on Directory condition

## 4. Policy Management

### 4.1 Policy Structure [S1, S4]

A Symantec DLP policy consists of:

| Component | Description |
|-----------|-------------|
| **Policy Name/Description** | Metadata identifying the policy |
| **Policy Group** | Assignment to a policy group for deployment targeting |
| **Detection Rules** | One or more rules defining match conditions |
| **Response Rules** | Actions to take when a violation occurs |
| **Exceptions** | Conditions that override/exclude certain matches |
| **Severity** | Incident severity level (1-4) |

**Rule types:**
- **Simple Rule**: Single match condition
- **Compound Rule**: Multiple conditions (all must match to trigger)

### 4.2 Policy Groups [S1, S4]

- Navigation: System > Servers and Detectors > Policy Groups
- Each policy belongs to exactly one policy group
- Default Policy Group deployed to all detection servers
- Custom policy groups can be deployed to specific detection servers
- Only users with Server Administration privilege can manage policy groups

**Creating a policy group:**
1. Navigate to System > Servers and Detectors > Policy Groups
2. Click Add
3. Provide name and description
4. Select target detection server(s)

### 4.3 Policy Templates [S1, S4]

- Pre-built XML files containing policy metadata, detection rules, group rules, and exceptions
- Navigation: Manage > Policies > Policy List > New Policy > Template List
- Compliance-oriented templates: GDPR, HIPAA, PCI DSS, SOX, GLBA
- Industry-specific templates
- Custom templates can be imported/exported
- Template import/export: Manage > Policies > Policy Template Import/Export

**Creating a policy from template:**
1. Select template from list
2. Choose data profile (if prompted)
3. Edit policy name/description (optional)
4. Select policy group
5. Edit rules/exceptions (optional)

### 4.4 Policy Detection Conditions [S1, S4]

Content-based conditions:
- Content Matches Exact Data (EDM)
- Content Matches Indexed Documents (IDM)
- Content Matches Keyword
- Content Matches Regular Expression
- Content Matches Data Identifier
- Content Matches VML Profile
- Content Matches MIP Tag Rule (16.0+)

Context-based conditions:
- Sender/Recipient Matches Pattern
- Sender/User Based on Directory Server Group
- Endpoint events (copy, paste, print, screen capture, USB)
- File properties (name, type, size, creation date)
- Protocol (SMTP, HTTP, FTP, IM)

### 4.5 Policy Exceptions [S1, S4]

- Added via Policy > Detection tab > Add Exception
- Can whitelist by sender/recipient pattern
- Email address patterns (no regex, no wildcard in Email/URL fields)
- URL domain patterns
- IP address patterns (regex/wildcard supported)
- Field limitation: 512 characters for domain field
- Global SMTP filtering: System > Settings > Protocol > SMTP > Filtering > L7 recipient filter

## 5. Response Rules

### 5.1 Response Rule Types [S1, S4]

| Type | Trigger | Use Case |
|------|---------|----------|
| **Automated Response** | Triggers automatically on incident detection | Default response type; conditions control execution |
| **Smart Response** | Manually triggered by authorized DLP user | Human remediation; false positive dismissal |

Smart Response limitations: no triggering conditions; limited actions (note, log, email, status)

### 5.2 Response Rule Actions by Detection Server [S1, S2, S3, S4]

**All Detection Servers:**
- Log to a Syslog Server
- Set Status
- Set Attribute
- Send Email Notification
- Limit Incident Data Retention

**Network Prevent for Email:**
- Block Message
- Modify Message (add/remove headers, redirect)
- Quarantine Message (via SMG integration)
- Add Header (X-header for downstream processing)
- Encrypt (via email gateway)

**Network Prevent for Web:**
- Block (ICAP response to proxy)
- Allow
- Content removal (remove sensitive HTML content)

**Endpoint Prevent:**
- Block (block data transfer + optional user notification)
- Notify (display popup notification to user)
- Encrypt (encrypt file via Endpoint Encryption integration)
- FlexResponse (custom plug-in actions)
- User Cancel (allow user to cancel blocked action with justification)

**Network Discover/Protect (Data at Rest):**
- Quarantine file
- Copy file
- Apply encryption
- Apply Digital Rights Management (DRM)
- Perform DRM on Data-at-Rest

**Cloud Applications / API Detection:**
- Add two-factor authentication
- Quarantine
- Block sharing
- Notify
- Apply classification label

**FlexResponse (Server and Endpoint):**
- Custom plug-in actions (Java JAR files)
- Server FlexResponse: configured via Plugins.properties file
- Endpoint FlexResponse: deployed to each endpoint
- Examples: email quarantine connect, encryption, DRM, redaction

### 5.3 Response Rule Conditions [S1, S4]

Conditions that gate response rule execution:
- Severity level
- Policy match
- Detection server type
- Protocol
- Sender/recipient patterns
- File properties

### 5.4 Response Rule Action Priority [S1, S4]

Multiple response rule actions can be configured, with each action evaluated independently. When multiple rules trigger, action execution priority determines the final outcome.

## 6. Deployment Vectors

### 6.1 Network DLP [S1, S4, S9]

#### 6.1.1 Network Monitor
- Passive monitoring via network span/tap port
- Monitors SMTP, HTTP, FTP, IM traffic
- Does not block; generates incidents for review
- Supports all detection technologies

#### 6.1.2 Network Prevent for Email
- In-line SMTP inspection
- Integration with MTA (Postfix, Sendmail, Microsoft Exchange)
- Reflecting mode: message analyzed, returned with X-headers
- Integration with Symantec Messaging Gateway (SMG) for quarantine
- Actions: block, redirect, quarantine, add header, modify, encrypt

#### 6.1.3 Network Prevent for Web
- ICAP protocol integration with web proxies
- Supports Squid, Blue Coat, and other ICAP-compliant proxies
- Secure ICAP (TLS) supported
- Performance tuning: match proxy concurrent ICAP connections to Web Prevent settings
- Actions: block, allow, content removal

#### 6.1.4 Network Discover
- Scans data at rest on network resources
- Supported targets:
  - File shares (CIFS, NFS, DFS)
  - Windows, Linux, AIX, Solaris file systems
  - Lotus Notes databases
  - SQL databases
  - Microsoft Exchange servers
  - Microsoft SharePoint servers
- Recognizes 330+ file types by binary signature
- High Speed Discovery: up to 1 TB/hour throughput (DLP 16+)
- Incremental scanning: only new/modified files
- Navigation: Manage > Discover > Discover Targets

#### 6.1.5 Network Protect
- Automated remediation for files found by Network Discover
- Actions: quarantine, copy, encrypt, apply DRM
- Replaces original files with marker/tombstone file
- Quarantine file restoration capability

### 6.2 Endpoint DLP [S1, S2, S4]

#### 6.2.1 Endpoint Prevent
Monitors and controls data channels on endpoints:

| Channel | Windows | macOS | Linux |
|---------|---------|-------|-------|
| Clipboard (copy/paste) | Yes | Yes | -- |
| Print/Fax | Yes | -- | -- |
| Removable Storage (USB) | Yes | Yes | -- |
| Local Drive | Yes | Yes | -- |
| Network Share | Yes | Yes | -- |
| HTTP/HTTPS | Yes | Yes | -- |
| FTP | Yes | -- | -- |
| Email (Outlook) | Yes | Yes | -- |
| Email (Lotus Notes) | Yes | -- | -- |
| Cloud File Sync (Box, Dropbox, Google Drive, OneDrive, iCloud) | Yes | -- | -- |
| Screen Capture | Partial | -- | -- |

**Agent Configuration:**
- Navigation: System > Agents > Agent Configuration
- Advanced Agent Settings available
- Agent installation packages generated from Enforce console
- Deployment: install_agent.bat (Windows), installer packages (macOS/Linux)
- Password required for Endpoint Tools administration
- Verification: System > Agents > Overview

#### 6.2.2 Endpoint Discover
- Scans local drives for policy violations
- Does NOT scan network-mounted drives or removable media
- Cannot be scheduled (manual start/stop only)
- Supports incremental scanning
- Platforms: Windows, macOS, Linux

#### 6.2.3 Agent Groups
- Group agents by AD Security Group, OU, or custom criteria
- Assign agent configurations per group
- Navigation: System > Agents > Agent Groups

### 6.3 Cloud DLP [S1, S2, S11, S24]

#### 6.3.1 Cloud Detection Service (CDS)
- Hosted detection service for email, web, and CASB
- No on-premises appliance required for cloud scanning
- Reduces latency and network traffic
- Supports DIM (Data in Motion) and DAR (Data at Rest)
- Actions: block, encrypt, quarantine, redirect

#### 6.3.2 Distributed Detection Service (DDS)
- Self-hosted DLP detectors in own infrastructure or cloud providers
- Unlike CDS, organization controls the detection infrastructure
- Same detection capabilities as CDS

#### 6.3.3 CloudSOC / CASB Integration
- Manages DLP policies for CASB data channel from single console
- Supports 100+ cloud applications (Office 365, G-Suite, Box, Dropbox, Salesforce)
- Cloud DLP Email policies
- Cloud DLP SWG (Secure Web Gateway) policies
- Directory Group Matching (DGM) for cloud policies
- EDM and IDM index support for cloud profiles
- Remote Indexer Tool for creating cloud-compatible indexes

#### 6.3.4 Cloud Storage Scanning
- Box Business and Enterprise accounts
- Dropbox
- Google Drive
- Microsoft OneDrive
- On-premises or cloud-based deployment options

## 7. Incident Management

### 7.1 Incident Lifecycle [S1, S4]

1. **Detection**: Content inspection triggers policy violation
2. **Incident Creation**: System creates incident record with metadata
3. **Classification**: Severity assignment (1-4: High, Medium, Low, Informational)
4. **Status Assignment**: Default statuses include New, Escalated, In Process, False Positive, Configuration Error, Resolved
5. **Investigation**: Assigned to queue/user for review
6. **Remediation**: Response actions applied (manual or automated)
7. **Resolution**: Incident closed with reason/resolution

### 7.2 Incident Attributes [S1, S4]

Standard attributes:
- Incident ID, severity, status, policy name, detection server
- Match count, match details
- Sender, recipient, protocol, file name
- Endpoint user, hostname, source IP
- Timestamp

Custom attributes:
- Defined at System > Incident Data > Attributes > Custom Attributes tab
- Populated via lookup plugins or manual entry
- Used in reporting and workflow automation

### 7.3 Incident Workflows (DLP 26.1) [S3]

New feature in DLP 26.1:
- Schedule and automate tasks throughout incident lifecycle
- Configure automated processing steps
- Handle escalation, assignment, notification patterns

### 7.4 End User Remediation [S1, S2]

- Built-in solution for decentralized incident remediation
- Integration with ServiceNow
- Configuration in Enforce console:
  - Select incidents for remediation
  - Define available incident information
  - Define available remediation actions
- Handles "Out of Office" remediators
- Automatic timeout/escalation if no action taken
- Can auto-enroll policy violators in training

### 7.5 Lookup Plugins [S1, S4]

Retrieve incident-related data from external sources to enrich incidents:

| Plugin Type | Purpose |
|-------------|---------|
| LDAP Lookup | Query Active Directory / LDAP for user attributes |
| Script Lookup | Execute custom scripts for data enrichment |
| Custom Plugins | Extensible plugin framework |

**LDAP Plugin Configuration:**
1. Create directory connection: System > Settings > Directory Connections
2. Create plugin: System > Lookup Plugins > New Plugin > LDAP
3. Map attributes: `attr.CustomAttributeName = searchbase:(searchfilter=$variable$):ldapAttribute`
4. Custom attribute names must match System > Incident Data > Attributes > Custom Attributes

## 9. Administration

**Role configuration:**
- Navigation: System > Login Management > Roles (or Manage > Roles depending on version)
- Each user assigned to one or more roles
- If multiple roles: user selects role at login (enhanced in 26.1 with Role Selection list)

**Role privileges govern:**

- Policy authoring rights

- Access to masked data

### 9.2 Authentication [S1, S4]

| Method | Configuration |
|--------|--------------|

| LDAP | System > Settings > Directory Connections |

### 9.3 Directory Connections [S1, S4]

- Navigation: System > Settings > Directory Connections

## 14. Key Architectural Findings

### 14.1 Critical Architecture Decisions

1. **Oracle dependency**: Enforce Server requires Oracle database (or embedded for small deployments). This is a significant infrastructure requirement.

2. **Policy-group-based deployment**: Policies are deployed via policy groups to specific detection servers. Each policy belongs to exactly one group. This creates a policy-to-server mapping model.

3. **15-minute agent sync interval**: DLP Agents check in every 15 minutes. Policy and configuration changes take up to 15 minutes to propagate to endpoints.

4. **SOAP API deprecated in 16.0**: All API integrations must use REST API for DLP 16.0+.

5. **Version gap (16.x to 25.x)**: Broadcom acquisition caused version renumbering. Migration path documentation is important.

6. **Hub-and-spoke model**: Single Enforce Server is a potential single point of failure. HA requires Veritas Cluster Server or similar.

7. **FlexResponse extensibility**: Both server-side and endpoint-side plugin architecture allows custom remediation actions, making the response system highly extensible.

8. **Cloud-native evolution**: CDS, DDS, and CloudSOC integration show the product evolving from on-premises-only to hybrid cloud deployment.

### 14.2 Integration Architecture Patterns

- **ICAP pattern**: Web proxy -> ICAP request -> DLP inspection -> ICAP response (block/allow)
- **MTA reflection pattern**: MTA -> DLP Email Prevent -> MTA with X-headers -> SMG enforcement
- **Agent-to-server pattern**: DLP Agent -> Endpoint Server -> Enforce Server (15-min poll)
- **Cloud detection pattern**: Cloud app -> CDS/DDS -> Policy evaluation -> Remediation action
- **SIEM forwarding pattern**: Enforce Server -> Syslog -> SIEM (incidents + audit logs)
